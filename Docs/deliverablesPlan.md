# Final Deliverables Plan (Post-Phase 7)

Once Phase 7 (Orchestration & Hardening) is complete, the engineering phase of the project concludes. The following steps outline how we will generate and compile the final deliverables required for submission.

## 1. Working Prototype Link or ≤3-min Demo Video
**Objective:** Provide a clear, visual demonstration of the system running end-to-end.
**Action Plan:**
- **Demo Video:** We will record a screen capture (using a tool like Loom, OBS, or built-in OS tools) demonstrating the following flow:
  1. Running the `pulse run --product groww` command in the terminal.
  2. Showing the terminal output as it processes ingestion, clustering, and summarization.
  3. Opening the live Google Doc to show the newly appended section appearing automatically.
  4. Opening the Gmail inbox to show the successfully created Draft containing the report and deep link.
- **Prototype Link:** We will include the GitHub repository link (if public) or a ZIP file, along with the live link to the Render-hosted MCP server (for reference, though credentials won't be exposed).

## 2. Latest One-Page Weekly Note (PDF/Doc/MD)
**Objective:** Provide the actual output generated by the AI for review.
**Action Plan:**
- After running the final end-to-end test for a selected product (e.g., Groww) for the current week, we will go to the generated Google Doc.
- We will export that specific weekly section as a PDF file.
- Alternatively, we will provide the raw `email.html` or rendered Markdown artifact generated by Phase 4.

## 3. Email Draft (Screenshot or Text)
**Objective:** Show the formatting and content of the stakeholder email.
**Action Plan:**
- We will open the generated draft in the Gmail web interface.
- We will take a high-quality screenshot showing the Subject Line, formatting (bullet points), and the clickable deep link.
- We will include this screenshot alongside the raw `email.txt` text output in the final submission folder.

## 4. Reviews CSV Used (Sample/Redacted)
**Objective:** Provide transparency into the raw data the LLM used to generate the report.
**Action Plan:**
- The system currently stores raw, PII-scrubbed reviews in the local SQLite database (`pulse.db`).
- We will write a tiny one-off script or use a sqlite-to-csv tool to dump the `reviews` table for the specific run into a `reviews_sample.csv` file.
- Because the ingestion pipeline already scrubs PII (emails, phone numbers, Aadhaar), the data is safe to share as-is.

## 5. README Documentation
**Objective:** Provide clear instructions for evaluators to understand and operate the project.
**Action Plan:**
We will update the repository's main `README.md` to include:
- **How to re-run for a new week:** Clear CLI instructions (e.g., `uv run pulse run --product groww --weeks 2`) and how to set up the necessary `.env` files if they wish to run it themselves.
- **Theme Legend:** A brief explanation of the architecture, specifically how themes are generated (UMAP + HDBSCAN clustering -> LLM Summarization) and what the theme metrics (review count, sentiment weight) mean.

---
**Timeline:** This plan will be executed immediately following the successful completion and testing of the Phase 7 orchestrator.
