# ProductReviewPulse — Project Review #2

## Overview
This is the second and final project review for the ProductReviewPulse agent. Following the initial architectural audit and edge-case assessment, a targeted stabilization phase was executed to address all identified gaps, implement the final Phase 7 orchestration features, and harden the system for production deployment.

## 1. Resolved Issues & Hardening

### Priority 0 (Critical Path)
- ✅ **Fixed the Status Tracking Bug (E24):** The orchestrator (`__main__.py`) no longer incorrectly labels a run as `published` if both the Google Docs and Gmail MCP calls fail. It now explicitly tracks the success of each target and only advances the run status if at least one publish vector succeeds. If both fail, the pipeline exits gracefully with a clear error.
- ✅ **Fixed Production Recipients Guard (E26):** Production deployments now strictly enforce that the `recipients.to` array in `products.yaml` is populated. If empty while `PULSE_ENV=production`, the pipeline crashes immediately instead of silently emailing a dummy/fallback address.

### Priority 1 (Resilience & Edge Cases)
- ✅ **Clustering Guardrail (E9):** `agent/clustering/pipeline.py` now includes a strict guard against extremely low-volume weeks. If fewer than 5 valid reviews are ingested, the clustering phase bypasses UMAP/HDBSCAN completely and returns a 0-cluster state, preventing mathematical crashes.
- ✅ **MCP Call Retries & Idempotency Upgrades (E22, E23):** 
  - `docs_ops.py` now implements a 3-attempt retry loop specifically when fetching the Google Doc to check for idempotency. This allows the system to survive transient timeouts (e.g., Render container cold starts).
  - It also handles the scenario where a Google Doc was manually deleted by the user but its ID is still cached in SQLite, gracefully falling back to re-creating the document.
- ✅ **Audit Data Formatting (Issue #2):** The JSONL audit trail bug in `__main__.py` was corrected, writing standard `\n` instead of escaped literals.

### Priority 2 (Orchestration & Phase 7)
- ✅ **GitHub Actions Weekly Cron (Gap #4):** `weekly-pulse.yml` has been added. It runs unattended every Monday at 07:00 IST via a matrix strategy covering configured products (Groww, INDMoney, Kuvera).
- ✅ **Deep Link Precision (Gap #3):** The Google Docs MCP integration now re-fetches the document immediately after appending the new weekly section to extract the exact `headingId` generated by Google. This allows the Gmail draft to include a precise `#heading=xxx` deep link, dropping stakeholders exactly where they need to be.
- ✅ **Strict Cost Cap Handling (E15):** The orchestrator now specifically catches the `PulseCostExceeded` exception, logs the financial metrics (spent vs. cap), and correctly marks the run status as `failed` rather than `pending`.

## 2. Code Quality & Architectural Refinements

- **App Store Ingestion Helper:** Refactored `agent/ingestion/appstore.py` to extract `_process_review` and `_parse_date`. The previously duplicated logic for PII-scrubbing, date parsing, and ID generation between the RSS feed and the HTML fallback scraper is now unified.
- **Play Store Pagination:** Fixed a bug in `agent/ingestion/playstore.py` where the scraper failed to `break` when encountering older reviews, resulting in excessive unnecessary HTTP calls.
- **Connection Management:** Hardened `agent/summarization.py` and `agent/storage.py` by wrapping SQLite connections in `contextlib.closing`, ensuring no file descriptor leaks occur during LLM long-polling. Added `PRAGMA busy_timeout=5000` to prevent `database is locked` errors during concurrent access.
- **LLM Hallucination Validation:** Added explicit guardrails in `summarization.py` to coerce invalid LLM sentiment output into standard values (`positive`, `mixed`, `negative`), preventing Pydantic failures later in the pipeline.
