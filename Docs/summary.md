# Project Summary: ProductReviewPulse

This document provides a comprehensive summary of the project architecture, implementation details, and current status.

## 1. Project Objective
**ProductReviewPulse** is an automated AI agent designed to bridge the gap between public customer feedback and product leadership. It ingests reviews from the Apple App Store and Google Play Store, uses LLMs to distill them into actionable themes, and delivers a premium "Weekly Pulse" report via Google Docs and Gmail.

---

## 2. Core Pipeline (Phases 1–4)
The pipeline is designed as a series of idempotent "stages," ensuring that if a process fails, it can be resumed without duplicating data.

*   **Phase 1: Ingestion & Scrubbing**
    *   **How:** We use `itunes-iap` for the App Store and `google-play-scraper` for the Play Store.
    *   **Security:** A PII Scrubber removes emails, phone numbers, and sensitive IDs before data hits the database.
    *   **Storage:** Reviews are stored in a local SQLite database (`pulse.db`) with a stable ID generated by `sha1(source + external_id)`.
*   **Phase 2: Clustering (The "Brains")**
    *   **How:** We use local embeddings (**bge-small-en-v1.5**) to convert review text into vectors.
    *   **Logic:** We apply **UMAP** for dimensionality reduction and **HDBSCAN** for density-based clustering. This automatically groups similar feedback (e.g., "app crashes on login") without needing pre-defined categories.
*   **Phase 3: LLM Summarization**
    *   **How:** We send the "medoid" (most representative) reviews of each cluster to a Groq-hosted **Llama 3** model.
    *   **Details:** The LLM generates a theme name, a summary, and "Action Ideas." We also implement a **Verbatim Validator** that ensures any quote included in the report actually exists in the raw data.
*   **Phase 4: Rendering**
    *   **How:** We use Jinja2 templates to convert the `PulseSummary` into:
        1.  **Google Docs Requests:** A sequence of `batchUpdate` commands (JSON).
        2.  **Gmail Content:** Clean HTML and plain-text emails.

## 3. The Delivery Engine (Phases 5 & 6)
This is where the agent interacts with the world. We use the **Model Context Protocol (MCP)** to ensure the agent never has direct access to Google credentials.

### A. The MCP Server (Infrastructure)
*   **Architecture:** A custom FastAPI server hosted on **Render**.
*   **Security:** Google `credentials.json` and `token.json` are stored as Render secrets. The agent only talks to the server via a secure URL (`MCP_SERVER_URL`).
*   **Approval Layer:** We implemented an `AUTO_APPROVE` toggle. In local development, the server asks for manual "y/n" confirmation; in production, it executes automatically.

### B. Google Docs Integration (Phase 5)
*   **Local State Management:** To avoid expensive "Search Drive" calls, we updated the `products` table in SQLite to store the `gdoc_id`. The agent checks the DB first; if missing, it creates a new Doc and saves the ID.
*   **Idempotency (The "Anchor" System):** Every report section starts with a hidden anchor string like `[pulse-groww-2026-W17]`. Before appending, the agent reads the Doc. If it sees that week's anchor, it skips the update to prevent duplicate sections.
*   **Batch Updates:** We use the `batchUpdate` API to insert text, apply formatting (Bold Heading 1, Bullet points), and create tables for the "Who This Helps" section.

### C. Gmail Integration (Phase 6)
*   **Draft-First Safety:** The agent creates a **Draft** instead of sending directly. This allows a human to review the pulse before it goes to stakeholders.
*   **CONFIRM_SEND Logic:** Sending is only enabled if the environment variable `CONFIRM_SEND=true` is set.
*   **Run Tracking:** We inject a custom header `X-Pulse-Run-Id` into every email. The agent searches for this header before acting—if a draft for that specific run already exists, it won't create another one.

---

## 4. State Management (SQLite Schema)
The database is the "Source of Truth" for the entire system:
*   **`products`**: Stores metadata and the cached `gdoc_id`.
*   **`runs`**: Tracks the status of every weekly run (`pending` → `ingested` → ... → `published`).
*   **`themes`**: Stores the LLM-generated insights and validated quotes.
*   **`reviews`**: Stores the PII-scrubbed raw feedback.
